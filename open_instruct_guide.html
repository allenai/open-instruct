🧠 What Is This Repo?

Open-instruct is AI2's production post-training codebase for instruction-tuning open-source LLMs. It's the implementation behind the TÜLU family of models and papers (TÜLU 1, 2, 3).

It supports three training paradigms, each building on the previous:

SFT

Supervised Fine-Tuning. Start from a pretrained base. Train on (prompt, response) pairs.

SFT

DPO

Direct Preference Optimization. Train on (prompt, chosen, rejected) triples.

DPO

GRPO

Online RL with verifiable rewards. Model generates its own training data each step.

GRPO
Research + Production
The codebase is research-grade but production-deployed. This tension creates accumulated complexity — multiple algorithms, model families, and infrastructure approaches coexist. Understanding this framing helps when you encounter seemingly redundant code.

Supported Models

Llama 3.x, OLMo / OLMo 2 / Olmo 3, Qwen. Chat templates exist for each. Adding a new model family means adding a tokenizer config + chat template.

⚡ Two Stacks: Old vs New

This is the single most important architectural fact. There are two implementations for SFT and DPO. When you read code, the first thing to figure out is which stack you're in.

OLD Accelerate / DeepSpeed

  • finetune.py — SFT
  • dpo_tune_cache.py — DPO
  • Uses HuggingFace Accelerate + DeepSpeed
  • Config: one flat FlatArguments dataclass
  • Supports LoRA / QLoRA (PEFT)
  • Still used for parameter-efficient experiments

NEW OLMo-core (FSDP)

  • olmo_core_finetune.py — SFT
  • dpo.py — DPO
  • grpo.py / grpo_fast.py — GRPO
  • Config: hierarchical composed dataclasses
  • FSDP + HSDP for multi-node
  • Active development happens here
Why Two Stacks Exist
OLMo-core was adopted mid-life for better FSDP semantics, hierarchical sharding (HSDP) for multi-node runs, and torch.compile support. The old Accelerate/DeepSpeed stack wasn't removed because LoRA workflows and some legacy configs still depend on it. Don't try to unify them without understanding what depends on each.

⚙️ Configuration System

The new stack uses hierarchical dataclasses. This is the most important pattern to internalize for reading and reviewing code.

# Each algorithm's config inherits from specialized sub-configs
@dataclass
class DPOExperimentConfig(
    ExperimentConfig,   # exp_name, seed
    ModelConfig,        # model_name_or_path, revision, attn_implementation
    TrainingConfig,     # lr, batch_size, max_steps, warmup_ratio
    DatasetConfig,      # dataset_mixer_list, transform_fn, max_seq_length
    LoggingConfig,      # wandb_project, logging_steps
    CheckpointConfig,   # save_every_steps, resume_from
    DPOConfig,          # loss_type, beta, label_smoothing
):
    pass

When reviewing a new feature, check that new config fields are added to the correct sub-config class — not just thrown into ExperimentConfig.

Parsing

ArgumentParserPlus (in utils.py) can parse multiple dataclasses simultaneously from CLI args. It handles JSON-string fields (dicts passed on the command line).

parser = ArgumentParserPlus((DPOExperimentConfig, TokenizerConfig))
args, tc = parser.parse()

📖 SFT: Supervised Fine-Tuning

SFT NEW olmo_core_finetune.py — primary, use for new work

SFT OLD finetune.py — Accelerate/DeepSpeed, use for LoRA/QLoRA

How It Works

1
Load pretrained model + tokenizer

Via model_utils.py. Applies chat template from TokenizerConfig.

2
Tokenize + cache datasets

Chat template applied, responses tokenized, prompt tokens masked to -100. Cached to disk by config hash.

3
Mix datasets + shard by DP rank

Each GPU gets different data. Multiple datasets blended by weight.

4
Training loop

Forward pass → language modeling cross-entropy loss (only on response tokens) → backward → step.

5
Checkpoint + optionally push to Hub

Saved in HF format. Can auto-push to HuggingFace Hub.

Critical Details

Chat Template Correctness
Getting the chat template wrong silently trains on wrong labels. The system uses named templates: tulu, olmo, llama, qwen. Each has different BOS/EOS behavior. When reviewing data code, verify the template name matches the model family.
Label Masking
Prompt tokens must be masked to -100 so the loss is only computed on response tokens. This happens during tokenization in dataset_transformation.py. A bug here (masking nothing, or masking everything) will train silently but produce a garbage model.
torch.compile
The OLMo-core version supports torch.compile. Changes to the model forward pass need to be compile-compatible — dynamic shapes, Python control flow inside the compiled region, and non-traceable ops will cause silent fallbacks or errors.

⚖️ DPO: Direct Preference Optimization

DPO NEW dpo.py — OLMo-core FSDP, primary

DPO OLD dpo_tune_cache.py — Accelerate/DeepSpeed, legacy

The Concept

DPO trains on (prompt, chosen response, rejected response) triples. The loss pushes the model toward preferred responses and away from rejected ones, without needing a separate reward model. It uses a reference model (a frozen copy of the starting policy) to stabilize training.

Loss Types (in dpo_utils.py)

TypeWhen To UseNotes
dpoDefaultOriginal paper. Uses reference model log-probs.
dpo_normVariable-length responsesAverages loss over tokens instead of summing. Better for datasets with long/short response mix.
simpoNo reference model availableUses length-normalized log-prob margin directly. No reference model needed.
wpoWeighted preferenceWeighted Preference Optimization variant.

Reference Logprob Caching — The Most Important Optimization

Why This Exists
DPO requires computing log p_ref(response) for every example on every training step. Running a full reference model forward pass each batch would double memory and compute. Instead, iterate the dataset once before training, save log p_ref per example to disk, and load from cache during training.
1
Pre-training: compute reference logprobs

Iterate entire dataset with reference model. Save per-example log-probs to a cache file (hash-addressed by config).

2
Training: load from cache, never re-run reference model

Each batch retrieves pre-computed reference log-probs. Memory savings: 2x (no reference model in GPU memory during training).

Cache Invalidation
If the reference model changes or the dataset changes, the cache must invalidate. This happens automatically via config hashing — but watch for bugs where a meaningful config change doesn't change the hash (e.g., a field that affects data but isn't included in the hash key).

Concatenated Forward

Chosen and rejected sequences are concatenated into one forward pass (not two). The loss is then split along the sequence dimension. This halves the number of forward passes per step. When reviewing changes to the DPO forward pass, make sure the split logic is correct.

🔄 GRPO: Online Reinforcement Learning

GRPO grpo_fast.py — primary, fast with packing + async

GRPO grpo.py — OLMo-core + Ray, cleaner but slower

The Concept

GRPO is online RL — the model generates its own training data at every step. It's the technique behind DeepSeek R1. The key idea: generate G responses per prompt, score each one, then compute advantages relative to the group (not globally).

The Training Loop

1
Sample prompts from dataset

Math problems, code tasks, instruction-following tasks, etc.

2
Generate G responses per prompt using vLLM

Typically G=8 or G=16. vLLM is used because it's 5-10x faster than model.generate().

3
Score each response with a verifier

Math: numeric equivalence. Code: execute and check tests. LM judge: another model scores it.

4
Compute group-relative advantage

Normalize scores within the G responses for each prompt. Mean=0, std=1. This is what makes it "group relative."

5
PPO-style policy gradient update

Clipped ratio of new policy vs old policy log-probs, weighted by advantage.

grpo_fast.py Specific Optimizations

OptimizationWhat It DoesWhy It Matters
Async RLHF Training and inference overlap via threads GPU never idle waiting for inference
Zero-gradient skipping If all G responses are correct OR all wrong, skip gradient Advantage = 0 for homogeneous groups → no useful gradient anyway
Sequence packing Same FlashAttention packing as SFT High GPU utilization for variable-length outputs

Verifiers — How Rewards Are Computed

All in ground_truth_utils.py. All implement a base class. All run asynchronously — synchronous verifiers block the training loop.

VerifierForHow
CodeVerifierCode tasksExecutes code in sandbox, checks test cases
MathVerifierMath tasksNumeric equivalence (handles formatting, LaTeX)
LMJudgeVerifierOpen-ended tasksAnother LLM judges via liteLLM (any provider)
RubricVerifierStructured tasksCustom rubrics in open_instruct/rubrics/
InstructionFollowingVerifierIFEval tasksConstraint checking (length, format, keywords)
MaxLengthVerifierLength controlPenalizes overly long responses
Adding a New Verifier
It must be async. A synchronous verifier will block the training loop and make GRPO training extremely slow. Follow the async pattern from existing verifiers.

Tool Use in GRPO

GRPO can train models that call tools (web search, code execution). The model generates output with embedded tool calls → environment pool executes them in sandboxes → results fed back → final response scored. Code: open_instruct/environments/ and open_instruct/environments/tools/.

Why vLLM and not model.generate()?

vLLM uses continuous batching and PagedAttention — it's 5-10x faster than HuggingFace's model.generate(). In online RL, generation is the bottleneck (you generate many times before each training update), so this speedup is critical.

💾 Data Pipeline

Two-Phase Design

Phase 1: Tokenization + Caching

Handled by dataset_transformation.py

  • Apply chat template (convert messages → tokens)
  • Truncate to max_seq_length
  • Mask prompt tokens to -100 (SFT)
  • Cache result to disk (content-addressed by config hash)

Phase 2: Loading

Handled by data_loader.py / dataset_transformation.py

  • Shard by DP rank (each GPU gets different data)
  • Collate: pad OR pack (FlashAttention packing)
  • Feed to training loop

Dataset Mixing

You rarely train on a single dataset. dataset_mixer_list takes alternating dataset_name, weight pairs:

# Example: 100% of tulu mixture + 50% sampling weight on another dataset
dataset_mixer_list: [
    "allenai/tulu-3-sft-mixture", "1.0",
    "some/other-dataset", "0.5"
]

Weights are relative — they control sampling probability. At epoch boundaries, the data reshuffles.

Content-Addressed Cache

The tokenized dataset cache is keyed by a hash of the tokenizer config + dataset config. Any meaningful change to these configs automatically invalidates the cache. Watch for bugs where a field affects output but isn't included in the hash.

Why Cache?
Tokenizing millions of examples takes hours. Caching means you only pay this cost once per unique config. Experiments that share a tokenizer and dataset config (e.g., different learning rates) reuse the same cache.

Key Dataset Columns by Algorithm

AlgorithmRequired Columns
SFTinput_ids, labels, attention_mask
DPOchosen_input_ids, chosen_attention_mask, rejected_input_ids, rejected_attention_mask
GRPOinput_ids (prompt only), env_config_key, ground truth column

📦 Sequence Packing (Padding-Free)

Without packing, most tokens in a batch are [PAD] — wasted compute. Packing eliminates this.

How It Works

Multiple sequences are concatenated end-to-end into one long sequence. FlashAttention's variable-length support (cu_seqlens) ensures attention doesn't cross sequence boundaries.

# Without packing (most tokens are PAD)
batch = [
    [token, token, PAD, PAD, PAD],
    [token, token, token, PAD, PAD],
]

# With packing (all tokens used)
packed = [token, token, EOS, token, token, token, EOS]
cu_seqlens = [0, 3, 7]  # sequence boundaries
position_ids = [0, 1, 2, 0, 1, 2, 3]  # positions reset per sequence

Implementation

In padding_free_collator.py:

  • TensorDataCollatorWithFlattening — for SFT packing
  • TensorDataCollatorWithFlatteningDPO — for DPO packing
Critical: cu_seqlens Must Be Correct
Any collator change that breaks cu_seqlens will silently produce wrong attention masks — sequences will attend across boundaries. The model will train, loss will decrease, but you're training on garbage. Always test with packing enabled when touching collation code.

Requirements

Packing requires FlashAttention 2. If FA2 isn't available, fall back to standard padding. Check attn_implementation config field.

mason.py Gotchas

--no_auto_dataset_cache Required on macOS
When calling mason.py locally on macOS, always pass --no_auto_dataset_cache (before the -- separator). Without it, mason.py tries to pre-cache the dataset locally, which fails because data_loader.py does import vllm at module level and vLLM doesn't install on macOS.
oe-eval-internal Required for Eval Launch
If your experiment uses --try_launch_beaker_eval_jobs_on_weka, the Docker image needs the oe-eval-internal directory. Clone it first: git clone --depth=1 https://github.com/allenai/oe-eval-internal.git oe-eval-internal

GPU Tests vs Training Experiments — Critical Distinction

PR CI checks for GPU_TESTS=EXPERIMENT_ID in the PR body and verifies the experiment passed. The experiment ID must come from running scripts/test/run_gpu_pytest.sh, not from a training run. These are completely different things.

📏 Code Conventions

Logging

Never use import logging directly
# WRONG
import logging
logging.info("training started")

# CORRECT
from open_instruct import logger_utils
logger = logger_utils.setup_logger(__name__)
logger.info("training started")
This gives consistent formatting with process rank, timestamps, and multi-node awareness.

Imports

  • All imports go at the top of the file, never inline (no conditional imports inside functions)
  • Use from package import module style, not import package.module

🔍 Review Patterns by Change Type

Data / Tokenization Changes

  • Does the config change affect the cache hash? (Meaningful config changes must invalidate cache)
  • Is the chat template name correct for the model family?
  • Are prompt tokens masked to -100 (only response tokens should contribute to loss)?
  • Does add_bos_token / add_eos_token match the model family's conventions?
  • Is dataset mixing weight logic correct (relative weights, not absolute)?

GRPO / RL Changes

  • Are new verifiers async? (Synchronous verifiers block the training loop)
  • Is advantage normalization per-group, not global?
  • Does zero-gradient skipping still work correctly (homogeneous groups)?
  • Are tool parsers handling the model output format correctly?

DPO Changes

  • If the reference model changes, does the logprob cache invalidate?
  • If the dataset changes, does the cache invalidate?
  • Is the concatenated forward pass split logic correct?
  • Is the new loss type handled in the loss-type dispatch?

Packing / Collation Changes

  • Is cu_seqlens computed correctly? (Off-by-one = wrong attention masks)
  • Are position_ids reset correctly at sequence boundaries?
  • Does it work both with and without packing enabled?
  • Has it been tested with FlashAttention 2?

Config Changes

  • Does the new field belong in the right config class?
  • Does the default value make sense for all experiments?
  • Is it added to CHANGELOG?
  • If it's a dataset/model field, does it affect the cache hash?

Performance Changes

  • Does torch.compile still work? (Dynamic shapes, non-traceable ops)
  • Is the change compatible with FSDP + HSDP?
  • Does gradient accumulation still work correctly?

💡 Why It's Built This Way

Understanding why decisions were made helps you make good decisions in the same spirit.

Why Two Training Stacks?

OLMo-core was adopted mid-life for better FSDP semantics, hierarchical sharding (HSDP) for multi-node, and torch.compile support. The Accelerate/DeepSpeed stack wasn't removed because LoRA/QLoRA workflows and some legacy configs still depend on it. Don't try to unify them without a full audit of what depends on each.

Why Cache Reference Logprobs?

Running a reference model forward pass every training batch would double GPU memory (two models simultaneously) and double compute. Pre-computing and caching eliminates the reference model from the training loop entirely. Trade-off: the cache can become stale if config changes aren't reflected in the hash.

Why vLLM for Generation?

In GRPO, generation is the bottleneck — you generate G responses for every prompt before each training step. vLLM's PagedAttention and continuous batching make it 5-10x faster than model.generate(). The complexity of integrating vLLM is worth it for training speed.

Why Sequence Packing?

Most real instruction datasets have highly variable-length responses. Without packing, a batch with one long response and many short ones wastes most of the batch. Packing fills every position with real tokens. The complexity cost (managing cu_seqlens, position_ids) is worth it for GPU utilization — especially at scale.

Why Content-Addressed Dataset Caching?

Tokenizing millions of examples takes hours. Without caching, every experiment restart retokenizes from scratch. Content-addressing ensures you get cache hits when running multiple experiments with the same data config (e.g., sweeping learning rates) and invalidation when data genuinely changes.

Why Hierarchical Config Dataclasses?

Early versions used flat argument parsers. As algorithms multiplied, so did arguments — leading to a single massive list of CLI args. Hierarchical dataclasses allow each concern (model, data, training, logging) to be defined once and composed. It also enables type-checking and auto-generated help text.

Why Ray for GRPO?

Online RL needs to run inference (vLLM) and training simultaneously on different GPU allocations. PyTorch's distributed primitives don't support heterogeneous workloads on the same cluster. Ray's actor model lets you allocate specific GPUs to specific roles and pass data between them asynchronously via queues.
