DPO vs GRPO

DPO — Direct Preference Optimization OFFLINE

The question DPO answers: Given a dataset of human preferences ("I prefer response A over response B for this prompt"), how do we fine-tune a model to produce preferred responses more often — without training a separate reward model?

The intuition: The model should assign relatively higher probability to chosen responses compared to the reference model. The reference model anchors what "normal" looks like, so we penalize drift, not just raw probability.

Core Insight
DPO doesn't train a reward model. It derives an optimal policy directly from preference labels using the fact that there's a closed-form relationship between the reward function and the optimal policy. The β (beta) parameter controls how much you trust the preferences vs. how much you want to stay close to the reference model. Higher β = stay close to reference.

How Data Is Structured

Every training example is a triple: one prompt with two responses — one preferred (chosen) and one not (rejected).

# What a raw preference example looks like
{
  "prompt": "What is 2+2?",
  "chosen": [
    {"role": "user",      "content": "What is 2+2?"},
    {"role": "assistant", "content": "It is 4."}
  ],
  "rejected": [
    {"role": "user",      "content": "What is 2+2?"},
    {"role": "assistant", "content": "I don't know."}
  ]
}

After tokenization (in dataset_transformation.py), each example becomes two separate full sequences — both containing the prompt tokens, just with different response tails:

KeyContainsShape
chosen_input_idsprompt tokens + chosen response tokens(1, seq_len)
chosen_labels-100 for prompt tokens, token IDs for response(1, seq_len)
chosen_attention_mask1 for real tokens, 0 for padding(1, seq_len)
rejected_input_idsprompt tokens + rejected response tokens(1, seq_len)
rejected_labels-100 for prompt tokens, token IDs for response(1, seq_len)

The -100 in labels is the PyTorch convention for "ignore this position in loss computation." It acts as a mask: the model predicts all tokens but we only measure loss on response tokens.

Forward Pass & Computing Logprobs

DPO needs a scalar per example: the sum of log-probabilities of the response tokens. Here's exactly how it's computed.

Step 1: Concatenate Chosen + Rejected (concatenated_forward)

Instead of two separate forward passes, both sequences are stacked along the batch dimension into one tensor of size 2*batch_size:

# open_instruct/dpo_utils.py: concatenated_inputs()
# batch has chosen_input_ids shape (B, seq_len) and rejected_input_ids shape (B, seq_len)
# After this, concatenated_input_ids has shape (2*B, max_len)

for k in batch:
    if k.startswith("chosen"):
        concatenated_batch[k.replace("chosen", "concatenated")] = pad_to_length(v, max_length)
for k in batch:
    if k.startswith("rejected"):
        concatenated_key = k.replace("rejected", "concatenated")
        concatenated_batch[concatenated_key] = torch.cat(
            (concatenated_batch[concatenated_key], pad_to_length(v, max_length)), dim=0
        )  # stacks chosen and rejected along batch dim → shape (2*B, max_len)

This halves the number of forward passes — one forward instead of two.

Step 2: Forward Pass → Logits

# The model sees a batch of 2*B sequences and produces logits
logits = model(**inputs).logits
# logits shape: (2*B, seq_len, vocab_size)
# logits[i][t] = unnormalized scores over vocabulary for position t of sequence i

Step 3: Per-Token Log-Probabilities (calculate_per_token_logps)

For each position, we want the log-probability of the actual next token (what actually appears in the sequence):

# open_instruct/padding_free_collator.py: calculate_per_token_logps()
# logits shape: (2*B, seq_len, vocab_size)
# labels shape: (2*B, seq_len)

# Convert to log-probabilities (logsoftmax over vocab dimension)
logp = F.log_softmax(logits, dim=-1)  # (2*B, seq_len, vocab_size)

# For each position, pick the log-prob of the label that actually appeared.
# logp[b][t][labels[b][t]] = log P(token at t+1 | tokens 0..t)
per_token_logps = torch.gather(logp, dim=2, index=labels.unsqueeze(-1)).squeeze(-1)
# Shape: (2*B, seq_len)
# per_token_logps[b][t] = log P(labels[b][t] | context)
The Off-by-One Trick
Language models are trained to predict the next token. So logits[t] predicts token at position t+1. The code slices per_token_logps[:, :-1] (dropping the last logit) and matches it against labels[:, 1:] (dropping the first label). This aligns each logit with the token it's predicting.

Step 4: Sum Over Response Tokens (_get_batch_logps)

# open_instruct/dpo_utils.py: _get_batch_logps()
per_token_logps = per_token_logps[:, :-1]  # shift: drop last logit
loss_mask = labels[:, 1:] != -100          # True only at response positions

if average_log_prob:
    # DPO-norm: average over response tokens
    return (per_token_logps * loss_mask).sum(-1) / loss_mask.sum(-1).clamp(min=1)
else:
    # DPO: sum over response tokens
    return (per_token_logps * loss_mask).sum(-1)

# Returns shape: (2*B,)  — one scalar per sequence

Step 5: Split Chosen and Rejected

# The first B values correspond to chosen, last B to rejected
all_logps = _get_batch_logps(per_token_logps, concatenated_labels)  # (2*B,)
chosen_logps   = all_logps[:bs]   # shape (B,) — one scalar per example
rejected_logps = all_logps[bs:]   # shape (B,) — one scalar per example

At this point we have two vectors, each of shape (B,): the total log-probability of the chosen response and the total log-probability of the rejected response, under the current policy.

The DPO Loss Function

Now the math. We have four scalars per example: policy_chosen_logps, policy_rejected_logps, ref_chosen_logps, ref_rejected_logps.

DPO Loss (per example)
π_logratios = log π_θ(chosen) - log π_θ(rejected) ← policy prefers chosen over rejected by how much? ref_logratios = log π_ref(chosen) - log π_ref(rejected) ← reference model prefers chosen over rejected by how much? logits = π_logratios - ref_logratios ← does the policy prefer chosen MORE than the reference does? loss = -logsigmoid(β × logits) ← binary cross-entropy loss. Loss is 0 when logits → +∞

This directly from the code:

# open_instruct/dpo_utils.py: dpo_loss()
pi_logratios  = policy_chosen_logps  - policy_rejected_logps
ref_logratios = reference_chosen_logps - reference_rejected_logps

logits = pi_logratios - ref_logratios  # "does policy prefer chosen more than ref?"

losses = -F.logsigmoid(beta * logits)   # main loss
         * (1 - label_smoothing)          # optional label smoothing
         - F.logsigmoid(-beta * logits)  # smoothing term (for rejected)
         * label_smoothing

# Rewards (detached — only for logging, not backprop)
chosen_rewards  = beta * (policy_chosen_logps  - reference_chosen_logps)
rejected_rewards = beta * (policy_rejected_logps - reference_rejected_logps)

Intuition for Each Term

TermMeaningGradient Effect
pi_logratios How much more the current policy prefers chosen over rejected Increasing this reduces loss
ref_logratios How much the reference prefers chosen over rejected Subtracted — acts as a baseline
logits = pi - ref Does the policy prefer chosen more than the reference does? Loss minimized when logits → +∞
-logsigmoid(β × logits) Binary cross-entropy: push logits positive Pull policy toward chosen, push away from rejected, relative to reference
β (beta) KL penalty strength — how much to stay near reference High β = small updates; low β = can diverge far from reference
Why Subtract the Reference?
Without subtracting ref_logratios, the model would just maximize log P(chosen) regardless of context. The reference term prevents "gaming": if the reference already strongly prefers chosen, the policy gets no gradient signal — it's already doing the right thing. If the reference also strongly prefers rejected (both are weird continuations), that's also fine. We only intervene when the policy is getting it wrong relative to the reference.

What Gets Backpropagated

The gradient flows through policy_chosen_logps and policy_rejected_logps, which are functions of the model weights. The reference logprobs are loaded from cache — no gradient flows through them.

# open_instruct/dpo_utils.py: compute_loss()
# Load reference logprobs from pre-computed cache (no grad)
ref_logps = reference_cache[batch["index"]]

return dpo_loss(
    policy_chosen_logps,          # ← gradient flows through this
    policy_rejected_logps,        # ← gradient flows through this
    ref_logps["chosen_logps"],   # ← just a constant from disk
    ref_logps["rejected_logps"], # ← just a constant from disk
    beta=args.beta,
)

Reference Logprob Cache

Before training begins, the codebase runs a full pass through the entire dataset with the reference model (frozen), computes chosen_logps and rejected_logps for every example, and stores them to disk indexed by the dataset row index.

# open_instruct/dpo_utils.py: build_reference_logprobs_cache()
chosen_tensor  = torch.full((full_dataset_size,), float("-inf"), ...)
rejected_tensor = torch.full((full_dataset_size,), float("-inf"), ...)

with torch.no_grad():
    for batch in dataloader:
        chosen_logps, rejected_logps, _ = forward_fn(model, batch, average_log_prob=...)

        # Store by dataset index — this is how we retrieve during training
        chosen_tensor[batch["index"]]  = chosen_logps
        rejected_tensor[batch["index"]] = rejected_logps

# All-reduce across distributed ranks (each rank processed different shards)
dist.all_reduce(chosen_tensor,  op=dist.ReduceOp.MAX)
dist.all_reduce(rejected_tensor, op=dist.ReduceOp.MAX)

# Save to disk — hash-addressed, so changing any config invalidates it
cache.to_disk(cache_path)

Cache Key (how invalidation works)

# open_instruct/dpo_utils.py: compute_reference_cache_hash()
config_str = json.dumps({
    "concatenated_forward": args.concatenated_forward,
    "dataset_config_hash": dataset_config_hash,  # ← changes if dataset changes
    "loss_type": args.loss_type,
    "max_train_samples": args.max_train_samples,
    "model_name_or_path": args.model_name_or_path,  # ← changes if model changes
    "model_revision": args.model_revision,
    "packing": args.packing,
    "use_lora": args.use_lora,
}, sort_keys=True)
hash = hashlib.sha256(config_str.encode()).hexdigest()[:16]

The cache lives at REFERENCE_LOGPROBS_CACHE_PATH (env var, defaults to a Weka path). During training, every batch lookup is just an index into this pre-computed tensor — O(1), no model forward pass.

Watch Out
If you add a new field that meaningfully changes the reference logprobs (e.g., a different prompt format) but don't add it to the hash, the old stale cache will be reused silently. Always add new config fields that affect tokenization or model behavior to compute_reference_cache_hash().

Loss Type Variants

DPO (dpo)

Sums log-probs over response tokens. Works well but sensitive to response length — long responses accumulate more signal regardless of quality.

logps = Σ log P(token_t)

DPO-norm (dpo_norm)

Averages log-probs over response tokens. More robust when chosen and rejected have very different lengths. This is the default in the codebase.

logps = (Σ log P(token_t)) / num_response_tokens

SimPO (simpo)

No reference model at all. Uses a margin γ/β between chosen and rejected. Simpler but less stable — the reference model serves as useful regularization.

logits = pi_logratios - γ/β
# no ref_logratios subtracted

WPO (wpo)

Weighted Preference Optimization. Weights the loss by how confident the current policy is in the response — high-confidence responses get higher weight.

weight = clamp(exp(logps_chosen + logps_rejected), max=1)
loss = standard_dpo_loss * weight

The dispatch is in dpo_utils.py: compute_loss(). SimPO doesn't use reference_cache at all — verify this when reviewing SimPO-related changes.

GRPO — Group Relative Policy Optimization ONLINE RL

The question GRPO answers: How do we train a model to solve tasks where the only feedback is "correct" or "incorrect" — and we don't have a pre-built dataset of preferred responses?

The intuition: Generate many responses to the same prompt. Score them. The model should learn to assign higher probability to responses that scored above the group average, and lower probability to those that scored below.

This is why it's called group relative: the advantage is computed relative to other responses for the same prompt, not globally across all prompts.

Why Group-Relative?
Different prompts have different difficulty. A score of 0.5 for a hard math problem is great; 0.5 for a simple instruction-following task is bad. Normalizing within each prompt's group removes this difficulty confound, giving cleaner gradient signal.

The Online RL Loop

Unlike DPO (which is offline — pre-computed dataset), GRPO generates its own data every step.

1
Sample prompts from dataset

num_unique_prompts_rollout prompts are drawn (e.g., 32 math problems).

2
Generate G responses per prompt using vLLM

num_samples_per_prompt_rollout responses per prompt (e.g., G=8). Total: 32×8 = 256 responses. vLLM is used because it's 5-10× faster than model.generate().

3
Score each response asynchronously

Verifiers (math check, code execution, LM judge) run on all 256 responses. Each response gets a scalar reward (e.g., 0.0 or 1.0 for correctness).

4
Compute group-relative advantages

For each prompt's G responses, normalize rewards: subtract group mean, divide by group std. Each response now has an advantage ∈ ℝ instead of a raw reward.

5
Assign advantage to every response token

Pack sequences. Every token in a response inherits the scalar advantage of the whole response.

6
Policy gradient update

Compute clipped importance-weighted loss. Backprop. Update weights. Sync weights back to vLLM engines.

After Group-Relative Normalization (scores → advantages)

# open_instruct/data_loader.py (DataPreparationActor)
scores = [1.0, 0.0, 0.0]   # raw rewards for [A, B, C]

scores_per_prompt = scores.reshape(-1, G)  # shape (1, 3): [[1.0, 0.0, 0.0]]

mean_grouped = scores_per_prompt.mean(axis=-1)     # [0.333]
mean_grouped = np.repeat(mean_grouped, G)             # [0.333, 0.333, 0.333]

std_grouped  = scores_per_prompt.std(axis=-1)      # [0.471]
std_grouped  = np.repeat(std_grouped, G)              # [0.471, 0.471, 0.471]

advantages = (scores - mean_grouped) / (std_grouped + 1e-8)
# A: (1.0 - 0.333) / 0.471 = +1.414  ← positive, model should do more of this
# B: (0.0 - 0.333) / 0.471 = -0.707  ← negative, model should do less of this
# C: (0.0 - 0.333) / 0.471 = -0.707  ← negative, model should do less of this

Assigning Advantage to Tokens

The scalar advantage for each response is broadcast to every response token. Prompt tokens get advantage 0 (they are masked and don't contribute to the gradient).

# Response A's tokens all get advantage +1.414
# Response B's tokens all get advantage -0.707
# Prompt tokens get 0 (masked out in the loss computation)

# This is implemented via the lookup array trick in data_loader.py:
lookup_advantages = np.zeros(len(advantages) + 1)  # index 0 → advantage 0 (for prompt)
lookup_advantages[1:] = advantages                  # index i+1 → advantage for response i

# response_masks values are 0 (prompt/pad) or i+1 (response i's tokens)
# So indexing lookup_advantages by response_masks gives the right advantage per token
packed_advantages = lookup_advantages[response_masks]

Group-Relative Advantage: The Math

Group-Relative Advantage Computation
G = num_samples_per_prompt (e.g., 8) r_i = reward for response i within the group μ_group = (1/G) Σ r_i ← group mean reward σ_group = sqrt((1/G) Σ (r_i - μ)²) ← group std reward A_i = (r_i - μ_group) / (σ_group + ε) ← normalized advantage

Zero-Gradient Skipping (grpo_fast.py optimization)

If all G responses for a prompt are identical in reward (e.g., all correct, or all wrong), then σ_group = 0, which means all advantages are 0, which means no gradient. The code detects this and skips those groups entirely:

# open_instruct/data_loader.py (DataPreparationActor)
if self.config.filter_zero_std_samples:
    # Filter out groups where all responses have same reward
    nonzero_std_mask = std_grouped_rewards > 0
    if nonzero_std_mask.sum() == 0:
        # All groups are homogeneous — skip entire batch, resample
        logger.info("all groups filtered (zero-std rewards); resampling")
        continue
Why Skip Zero-Std Groups?
When all responses to a prompt are equally wrong (or equally right), the advantage is 0 for every token — the gradient is literally zero. Computing the forward pass, backward pass, and optimizer step for zero gradient wastes compute. Skipping these groups speeds up training without losing information.

The Policy Gradient Loss

With advantages computed, the training step uses a clipped importance-weighted loss (like PPO, but without a value network).

Step 1: Compute Per-Token Log-Probs for the Current Policy

# open_instruct/grpo_fast.py (training loop)
local_logprobs_BT, entropy_BT = grpo_utils.forward_for_logprobs(
    self.model,
    data_BT.query_responses[i],   # packed query+response tokens
    position_ids=data_BT.position_ids[i],
    temperature=self.streaming_config.temperature,
)
# local_logprobs_BT shape: (batch, seq_len)
# Contains log P(token_t | tokens_0..t-1) for each response token

Step 2: Compute the Importance Ratio

Importance Sampling Ratio
ratio = π_θ(a_t | s_t) / π_old(a_t | s_t)       = exp(log π_θ(a_t) - log π_old(a_t))       = exp(new_logprobs - old_logprobs) ← per token 
# open_instruct/grpo_fast.py
logprobs_diff_BT = new_logprobs_BT - old_logprob_BT  # per token difference
ratio_BT = torch.exp(logprobs_diff_BT)                  # importance weight per token
# ratio > 1 means current policy assigns MORE probability to this token than old policy
# ratio < 1 means current policy assigns LESS probability

The "old logprobs" are the logprobs from vLLM at generation time (or from the policy at the start of the current training step if using multiple epochs).

Step 3: DAPO Clipped Loss (default loss_fn)

DAPO Policy Gradient Loss (per token)
pg_loss_1 = -A_i × ratio ← unclipped loss pg_loss_2 = -A_i × clip(ratio, 1-ε_low, 1+ε_high) ← clipped loss loss = max(pg_loss_1, pg_loss_2) ← conservative: take the worse of the two 
# open_instruct/grpo_utils.py: compute_grpo_loss()
if config.loss_fn == GRPOLossType.dapo:
    pg_losses  = -advantages * ratio
    pg_losses2 = -advantages * torch.clamp(ratio, 1.0 - config.clip_lower,
                                                     1.0 + config.clip_higher)
    clipfrac = (pg_losses2 > pg_losses).float()  # fraction of clipped tokens
    pg_loss   = torch.max(pg_losses, pg_losses2)  # take conservative (higher) loss

Intuition for the Clipping

If advantage is positive (good response), we want to increase the token's probability. But if ratio is already much > 1 (we've already increased it a lot), further increasing it is risky — we don't trust the gradient signal that much. The clip limits how aggressively we update.

If advantage is negative (bad response), we want to decrease probability. Same clip applies in the other direction.

Step 4: Optional KL Penalty Against Reference

# If load_ref_policy=True, add KL divergence penalty
ref_logprobs_diff = (new_logprobs - ref_logprobs).clamp(-40.0, 40.0)
kl_all = model_utils.estimate_kl(ref_logprobs_diff, ratio)
kl = kl_all[config.kl_estimator]  # e.g., "k3" estimator

per_token_loss_BT = pg_loss_BT + self.args.beta * kl_BT

Step 5: Masked Mean → Scalar Loss → Backward

# Average over response tokens only (prompt tokens masked)
loss = masked_mean(per_token_loss_BT, response_mask_BT, None, loss_denominator)

# Scale for distributed training (DeepSpeed averages over ranks; we want sum)
loss *= self.args.world_size // self.args.sequence_parallel_size

self.model.backward(loss)
if is_accumulation_boundary:
    self.model.step()  # optimizer step after gradient accumulation

Packing & Async in grpo_fast.py

Sequence Packing in GRPO

Each training step produces many variable-length responses. Without packing, shorter responses waste most of the batch with padding. The pack_sequences() function bins responses into packed batches:

# open_instruct/rl_utils.py: pack_sequences()
# Input: 256 (query, response) pairs of varying length
# Output: several packed batches, each up to pack_length tokens

for i, (query, response) in enumerate(zip(queries, responses)):
    query_response = query + response

    # If adding this seq would exceed pack_length, flush current pack
    if len(query_response) + len(cur_data) > effective_pack_length and len(cur_data) > 0:
        query_responses.append(cur_data)        # flush
        # ... reset cur_data, attention_mask, etc.

    cur_data.extend(query_response)

    # Attention mask: encodes which "document" (sequence) each token belongs to
    # Token at position t gets value (i+1) if it's from sequence i, else 0 (query/pad)
    cur_attention_mask.extend([i + 1 - offset for _ in range(len(query_response))])

The attention mask values encode document identity. reset_position_ids() uses this to generate correct position IDs that reset to 0 at each sequence boundary.

Async RLHF

In grpo_fast.py, training and inference overlap via threads. While the GPU is doing the backward pass for batch N, the CPU is preparing batch N+1 (calling vLLM, running verifiers). This hides inference latency behind training time:

# DataPreparationActor runs in a Ray remote actor (separate process)
# It prepares data asynchronously and puts it into a queue
# The training actor pulls from the queue, overlapping data prep with training

# The data prep actor does:
#   1. Sample prompts
#   2. Send to vLLM (async)
#   3. Run verifiers on responses (async)
#   4. Compute advantages
#   5. Pack sequences
#   6. Put into queue
#
# The training actor does:
#   1. Pull from queue (blocks until data ready)
#   2. Run training step (forward + backward + step)
#   3. Sync weights to vLLM
#   → Meanwhile, data prep is running step N+1 in parallel

Gradient Flow Comparison

DPO Gradient

Flows through the per-token log-softmax at every response token of both chosen and rejected. The gradient at token t pulls the logit for the correct token up (for chosen) or down (for rejected).

Strength proportional to: β × (1 - sigmoid(β × logits))

GRPO Gradient

Flows through per-token log-probabilities at every response token, scaled by the response's advantage. Positive advantage → increase token probability. Negative → decrease.

Strength proportional to: advantage × (ratio or clipped ratio)
