OPD: how it's built, and why

On-Policy Distillation adds a dense, token-level teacher signal to the existing OLMo-core GRPO loop. The student keeps generating its own rollouts; a frozen teacher scores the exact prefixes the student visited; the learner regresses the student toward the teacher's top-k distribution. This page is the mental model, the wiring, and the honest list of where it costs you.

1 The one-sentence thesis

If you remember one thing: OPD is dense supervision on on-policy states. GRPO asks “did the final answer earn reward?” (one sparse scalar per trajectory). OPD asks “at every token the student produced, what distribution would the teacher have preferred?” (a target at every position).

How it differs from the things it looks like:

MethodTrains on which states?Supervision
SFTFixed dataset trajectoriesHard token labels
Classic KDTeacher-generated trajectoriesTeacher distribution
RLVR / GRPOStudent-generated trajectoriesSparse outcome reward
OPDStudent-generated trajectoriesDense teacher distribution at each token

OPD = KD's dense signal applied to GRPO's on-policy state distribution. The "on-policy" part is what makes it different from ordinary KD: the teacher scores states the student actually visits, not states the teacher would have generated.

The objective in one line

# For each response token, minimize forward KL of teacher onto student:
loss = Σ_k p_teacher(k) · ( log p_teacher(k) − log p_student(k) )    # k over teacher's top-K tokens

GRPO + OPD:   total = pg_loss + beta·ref_kl + opd_loss_coef·opd_loss
Pure OPD:     total = opd_loss_coef·opd_loss          # task loss zeroed; rollout infra stays online

2 What we built on top of, and why

OPD did not introduce a new trainer. It is a hook inside the existing OLMo-core GRPO online pipeline. That single decision drives almost every structural choice below.

The pre-existing GRPO online machinery (unchanged)

Actor / pieceRole we reused
LLMRayActor student vLLMSamples current-policy rollouts (queries, responses, masks, vLLM logprobs).
DataPreparationActorAccumulates rollouts, computes rewards + GRPO advantages, packs sequences, collates per-worker batches. This is where OPD hooks in.
StreamingDataLoaderServes CollatedBatchData to the learners; also builds mock batches.
GRPOTrainModuleRecomputes student logits, applies the GRPO loss, steps the optimizer.

Why hook between rollout and packing?

The teacher needs to score prompt + student_response — which only exists after rollout but before the sequences are packed/masked. Inserting the teacher scoring exactly there means one batch contract serves all three modes (plain GRPO, GRPO+OPD, pure OPD). The teacher fields are just optional extra tensors riding the same PackedSequences → CollatedBatchData path.

What was genuinely new

  • A teacher scorer — a separate frozen vLLM engine wrapped in a Ray actor (opd_utils.py), using vLLM's prompt_logprobs to score (not generate).
  • DistillKit — a small reusable math layer (distillkit/) for the sparse signal + forward-KL loss, deliberately decoupled from GRPO so SFT/DPO/OPSD can reuse it later.
  • Optional teacher tensors threaded through the data model ([B, T, K] next to the existing [B, T]), including the sequence-parallel splitter.
  • A no-second-forward loss path in the learner: gather student top-k logprobs from the same forward that already computes GRPO logprobs.
  • Validation (opd_validation.py): hard guards for pure-OPD config and teacher/student vocab compatibility.

The teacher scores — it does not generate

Sampling params are top_k=-1, prompt_logprobs=K, max_tokens=1, detokenize=False, logprobs_mode="raw_logprobs". The single generated token is thrown away; all the value is in prompt_logprobs, i.e. the teacher's next-token distribution at every position of the supplied sequence. We feed the teacher pre-tokenized ids (the student's), so the teacher never re-tokenizes text.

Subtlety vLLM forces on us

vLLM sometimes emits an empty slot for the first prompt token (no distribution predicts position 0). extract_response_topk_from_prompt_logprobs detects this and accepts both alignments — full length P+R (response starts at prompt_len) or P+R−1 (response starts at prompt_len−1). The GPU test asserts the real response token actually appears in the returned teacher top-k, which is the end-to-end proof that alignment is right.

6 The loss math

The v1 loss is the unnormalized teacher-top-k forward KL. Two words matter: top-k (we never materialize full-vocab teacher logits) and unnormalized (we do not renormalize the teacher over the top-k set).

def forward_kl_topk_from_logprobs(student_topk_logprobs, teacher_signal):
    finite   = isfinite(teacher.logprobs)              # missing top-k slots are −inf
    p_teach  = where(finite, exp(teacher.logprobs), 0) # raw teacher probs, NOT renormalized
    topk_mass = p_teach.sum(-1)                         # diagnostic: how much mass top-k captured
    loss = Σ_k  p_teach · ( teacher.logprobs − student_topk_logprobs )
    return loss, topk_mass

Why top-k?

Full-vocab teacher logprobs at every response token would be B·T·V (V ≈ 150k). Top-k stores B·T·K (K=128). That's the difference between feasible and not, for an online loop.

Why unnormalized?

At T=1 the raw top-k logprobs are the teacher's true probabilities; the omitted tail is honestly omitted. teacher_topk_mass then means something — "is K big enough?" Renormalizing would peg that mass to ~1 and destroy the diagnostic. (A normalize_topk=True path exists for a future distillation-temperature feature, but it is not the default.)

7 Key design decisions (and the reasoning)

7.1 Temperature: GRPO tempered, OPD raw (T=1)

The single student forward produces two views of the logits. The sampled-token logprobs use logits ÷ rollout_temperature because GRPO's importance ratio must be evaluated under the policy that actually sampled. The OPD top-k logprobs use raw logits (T=1) because the teacher returns raw logprobs.

Why this matters

A KL is a divergence between two distributions; both sides must be at the same temperature or it isn't a coherent objective. Teacher is T=1, so the student OPD view must be T=1. The rollout temperature governs which states are visited (exploration) and the IS ratio — not the distillation target. Mixing them would silently train KL(teacher@T1 ‖ student@T0.7), biasing the student's true distribution flatter than the teacher. The sparse-top-k representation actually pins the distillation temperature to 1.0 (you can't properly temper a teacher you only have K logprobs for).

7.2 No second student forward

OPD reuses the forward that GRPO already runs. forward_for_logprobs_and_topk gathers the sampled-token logprob (for the ratio) and the teacher-top-k logprobs (for distillation) from the same logits. Cost is two log_softmax over the vocab (one tempered, one raw) instead of a whole extra forward pass.

7.3 Teacher is fed pre-tokenized ids → vocab check is id-level

Because we pass the student's token ids straight to the teacher (never re-tokenizing text), BPE merge-rule differences are irrelevant. The only thing that must hold is that teacher token id i means the same token as student id i. validate_teacher_student_tokenizer_compatibility checks exactly that (every teacher token → same student id, no in-range conflicts, matching bos/eos/unk), allowing student-only extra tokens above the teacher range (e.g. an added pad). It fails loud at startup instead of silently clamping bad ids.

7.4 Pure OPD is a hard contract, not a warning

validate_pure_opd_reference_config errors unless beta == 0 and load_ref_policy == false when opd_use_task_rewards=false. In pure OPD the task loss is zeroed (which also zeroes beta·ref_kl), so allowing a non-zero beta or a loaded ref policy would only waste reference-policy setup and mislead.

7.5 Sentinels, not special-casing

Query and padding positions carry sentinel teacher values (id 0, logprob −∞). In the loss, −∞ → exp → 0, so those positions contribute exactly zero and the response mask zeros them again. No branchy "is this a response token" logic in the hot path — the data is shaped so the math just works.

8 Bottlenecks — the honest list

This is what your colleague is really asking about. Ranked roughly by how likely it is to bite at scale.

BottleneckSeverityWhyWatch / mitigate
Teacher scoring on the critical path high Step 3 is a blocking ray.get inside data prep — packing can't start until the teacher returns. If the teacher is bigger/slower than the student (e.g. 4B teacher, 0.6B student), it can dominate the time-to-batch. time/opd_teacher_scoring; add teacher engines (opd_teacher_num_engines); rely on async_steps so data prep runs ahead of the learner.
Teacher GPU memory high The teacher is a full extra vLLM engine (weights + KV cache) competing with student rollout and the learner. In single_gpu_mode everything collocates — the debug scripts drop gpu_memory_utilization to 0.35 to fit. opd_teacher_gpu_memory_utilization, opd_teacher_tensor_parallel_size; give the teacher its own GPUs at scale.
prompt_logprobs=K over the whole sequence med vLLM computes and serializes top-K logprobs for every prompt position (prompt + response), but we only keep the response rows. K logprobs/position is real compute + Ray transfer cost; prompt rows are discarded. Keep prompts short where possible; K is a direct multiplier on transfer size.
[B, T, K] tensor payload med Teacher ids+logprobs are K× (=128×) the size of the existing [B, T] tensors. They ride through Ray queue serialization, pack_sequences (per-token torch.stack), collation, and SP split — CPU memory in data prep, GPU memory in the learner. opd_topk, packed batch size, response length; this is the main RAM/VRAM tax of OPD.
Two vocab log_softmax in the learner med-low With OPD on, the forward computes a tempered and a raw log_softmax over [B, T, V] (V ≈ 150k), plus a [B,T,K] gather. Cheaper than a second forward, but not free. Inherent to the no-second-forward design; acceptable tradeoff.
Async staleness (shared with GRPO) low* The rollout/teacher producer can run ahead of the learner; the teacher scored an older student policy. The ρ-correction machinery already handles train/infer mismatch for GRPO. num_steps_off_policy, val/rho_weight, val/rho_clipfrac.

* Low for OPD specifically, but worth naming because it's the same producer/consumer pipeline OPD plugs into.

The structural tradeoff, in one line

OPD buys a much denser learning signal (a target at every token vs one scalar per trajectory) at the cost of a second model in the loop (memory + a blocking scoring phase) and K× heavier batches. Whether it's worth it is the matched GRPO-only control that the design doc still flags as owed.

9 Extension points — why it's shaped for OPSD

The split between "online plumbing" (GRPO owns) and "sparse distillation math" (DistillKit owns) exists so the next variant doesn't require a rewrite.

Future variantWhat changesWhat stays
OPSD (self-distillation)Teacher prompt becomes demo-conditioned (teacher_prompt = problem + demonstration); teacher may be the student itself, weight-synced (self_sync).The scorer interface, the sparse signal, the forward-KL loss, the whole batch path.
PG OPD (reverse-KL reward)A sampled-token k1 estimator used as a token reward instead of direct backprop.Teacher scoring + extraction. (Explicitly not built — higher variance, discards most of the top-k signal.)
Offline SFT/DPO distillationScore fixed dataset trajectories instead of online rollouts.DistillKit losses/signals are mode-agnostic and importable anywhere.

This is why the loss lives in distillkit/ with no GRPO imports, and why the teacher prompt is a teacher_prompt_mode knob rather than hardcoded: v1 ships student_prompt, OPSD just adds demo_conditioned without touching the math.

10 Talking-points cheat-sheet

If your colleague asks, here are crisp answers.

"What is it, in one line?"
Dense token-level teacher supervision on the student's own on-policy rollouts, via teacher-top-k forward KL, bolted onto OLMo-core GRPO.
"Did you write a new trainer?"
No — it's a hook between rollout-accumulation and packing in DataPreparationActor, plus an extra optional tensor on the existing batch contract. Plain GRPO, GRPO+OPD, and pure OPD share one path.
"How does the teacher score?"
A frozen vLLM engine, fed the student's pre-tokenized query+response ids with prompt_logprobs=K. It scores, doesn't generate. We extract the response-token rows as [T, K].
"What's the trickiest part?"
Alignment: teacher distribution-for-token-t must match student prediction-of-token-t through packing + the causal shift-by-one. Sentinels on query/pad positions, [:, 1:] slice at train time, GPU test proves it.
"Biggest bottlenecks?"
(1) Teacher scoring is a blocking phase on the path to a batch; (2) the teacher is a second model eating GPU memory; (3) [B,T,K] batches are K× heavier through Ray/packing/collation.
"Any subtle correctness things you fixed?"
GRPO logprobs stay tempered (rollout T) for the IS ratio; OPD logprobs use raw T=1 to match the teacher — otherwise the KL is between mismatched temperatures. Plus hard-fail validation on pure-OPD config and teacher/student vocab id compatibility.
"Is it proven better than GRPO?"
Not yet. A 500-step Qwen3-0.6B/4B rehearsal proved the system path (teacher loads, scoring happens, tensors reach the learner, loss logs, training completes; top-k mass ≈ 1, sampled-in-top-k ≈ 1). A matched GRPO-only control is still owed before any quality claim.
